## 1. Architecture & Data Flow

### Before (Static, Hardcoded Architecture)

```
src/data/tools.ts          ──→  53 hardcoded tool objects (651 lines)
src/data/articles.ts       ──→  10 hardcoded article objects (756 lines)
src/data/categories.ts     ──→  16 hardcoded category objects (100 lines)
src/data/authors.ts        ──→  2 hardcoded author objects (69 lines)
src/data/*.ts              ──→  14 more hardcoded data files (categories, content, recipes, faq, pregnancy...)
src/consts.ts              ──→  Site brand, nav links, GA4 ID, SEO defaults (121 lines)
                              ↓
               Imported directly by every .astro page.
               No database. No admin UI.
               Every content change = code change = deploy.
```

### After (CMS-Driven Architecture)

```
PAYLOAD CMS BACKEND (F:\cloud\cms\)                    ASTRO FRONTEND (F:\cloud\)
┌─────────────────────────────────────────┐           ┌─────────────────────────┐
│  SQLite/Turso Database                  │           │  astro build             │
│  ├── tools: 53 rows                     │  REST     │     ↓                    │
│  ├── articles: 11 rows                  │◄─────────│  src/lib/cms.ts          │
│  ├── categories: 16 rows                │  GET      │     ↓                    │
│  ├── authors: 2 rows                    │           │  All .astro pages        │
│  ├── pages, redirects, globals...       │           │     ↓                    │
│  └── settings, social, ads...           │           │  getStaticPaths()        │
│                                         │           │  generates routes from   │
│  Payload Admin UI (/admin)              │           │  CMS data at build time  │
│  └── editors add/edit content here      │           │     ↓                    │
└─────────────────────────────────────────┘           │  Static HTML output      │
                                                      │  (Vercel deploy)         │
                                                      └─────────────────────────┘
```

**The bridge layer** (`src/lib/cms.ts`) is a single 714-line module that every page imports. It follows a **try-CMS-first-then-fallback** pattern:

1. On `astro build`, each page calls `getTools()`, `getArticles()`, `getCategories()`, etc.
2. These functions `fetch()` the Payload CMS REST API at `CMS_URL` (default `http://localhost:3000`)
3. **If CMS is reachable:** returns live database content
4. **If CMS is unreachable** (or `CMS_DISABLE=1`): falls back to the old `src/data/*.ts` files
5. This means the site still builds even if the CMS is down — the old static data acts as a safety net

## 2. Content Management

### Adding a New Article — Before (Old Way)

1. Open `src/data/articles.ts`
2. Add a new object at the end of the `ARTICLES` array with slug, title, body (HTML), FAQs, sources, SEO metadata
3. Open `src/data/tools.ts` to add related tool links
4. Open `src/data/authors.ts` if a new author
5. **Commit code → Deploy entire site**

### Adding a New Article — After (New Way)

1. Log into **CMS Admin** at `https://cms.healthylifestyles.com/admin`
2. Click **"Articles"** → **"Create New"**
3. Fill in: title, slug (auto-generated), body (rich text with block editor), SEO metadata, category, author
4. Click **"Publish"** → **Zero code changes**
5. On next `astro build`, the article appears automatically at `/wellness-hub/<slug>`

### Adding a New Tool — After

1. Log into CMS Admin → **"Tools"** → **"Create New"**
2. Fill in: name, slug, category, formula/calculator config, SEO metadata, description
3. Click **"Publish"** → **Zero code changes**
4. On next build, appears at `/tools/<slug>`

**Exception (11 tools):** Complex interactive calculators (meal planner, strength program builder, chronotype quiz, fasting timer, etc.) use custom Astro/React islands in `src/islands/`. These 11 tools require code changes to update their interactive UI, but their **metadata, category, SEO, and descriptions** are managed via CMS. The static routing for these 11 custom tools is handled by dedicated `.astro` files in `src/pages/tools/` (e.g., `meal-plan-generator.astro`).

## 4. Performance & Build

### Astro SSG Performance

The site uses `output: 'static'` with `@astrojs/vercel` adapter. Key performance guarantees:

1. **Build-time fetching:** All CMS calls happen during `astro build`, producing plain static HTML. The CMS is **not queried at runtime** by visitors.
2. **Zero runtime DB overhead:** After deploy, every page is a static file on Vercel's CDN. Response times are identical to the old hardcoded site.
3. **CMS can be offline:** Even if the CMS backend is down, `astro build` succeeds because `cms.ts` falls back to local `src/data/*` data.
4. **Build speed impact:** CMS fetches add ~10-30 seconds to build time (depending on network). Each fetch has a 4-second timeout.

### Caching Strategy

- **No application-level cache** — not needed for SSG (all content baked into static files)
- **CDN caching** — Vercel's Edge Network handles all caching automatically
- **ISR not used** — SSG builds are triggered on content publish (via CMS webhook or manual deploy)

## Summary

The migration from **static hardcoded data** to **Payload CMS** is complete and production-safe. The architecture uses a **hybrid pattern**: Astro SSG at build time fetches from the CMS via REST, with a transparent fallback to local static data. This combines the **editorial flexibility** of a headless CMS with the **performance** of static site generation. All URLs are preserved. No zombie files, no stale redirects, no duplicate fetching logic. The site is ready for production deployment.
